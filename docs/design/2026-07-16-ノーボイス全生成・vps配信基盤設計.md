## 0. 方針転換の決定事項（初期設計からの差分）

| 項目 | 旧 | 新 | 理由 |
|---|---|---|---|
| 声・ナレーション | 案①朗読が主軸 | **一切入れない**（生声・AI声とも） | 訴訟リスク極小化。声の権利・効能表現・多言語の壁が同時に消える |
| ニッチ | 睡眠固定 | **候補3ニッチをデータで検証して選ぶ**（§3） | 睡眠・作業BGMはレッドオーシャン確定（調査4.1） |
| 生成手段 | AI生成前提 | **プロシージャル第一・AIは条件付き併用**（§2） | 依拠性ゼロ・著作権主張可・Mac/VPSで完結 |
| インフラ | フェーズ0はMac完結 | **ローカルサーバー（Proxmox・稼働中）を主機に、LAN内GPUノードをHTTP API越しに利用**。配信もまずローカルサーバーから。VPS（保有済み）は自宅回線で不足する場合の移設先としてフェーズ4で判断（§7） | 常時稼働マシンが既にあるためMacスリープ問題・VPS前倒しの必要が消えた。`-c copy`配信ならCPUで足りる |
| TTS/朗読ステージ | パイプライン[2] | **削除** | — |

不変の原則（ホワイト運用・完全オリジナル・自社ライセンス・人間レビュー必須・効能断定なし）は維持。
むしろ調査の結果、**この原則が2026年のYouTube環境（inauthentic content摘発・AI slop取り締まり）への
正しい防御である**ことが裏付けられた。

## 1. 戦い方の再定義

**「声なし長尺アンビエント」はYPP審査却下率が最も高いジャンル群**（調査1.1）。
そこで戦略の中核を「音や映像の量産」ではなく、次の3点に置く:

1. **機能・物語・世界観のレイヤーを構造的に持つフォーマットを選ぶ**（ニッチ選定自体が防御。§3）
2. **動画ごとの固有性をパイプラインが機械的に保証・記録する**（uniqueness manifest。§6.1）
3. **収益の主軸を広告RPMに置かない**（素材ライセンス販売・Spotify配信・メンバーシップ等の
   非広告収益をニッチ選定の評価軸に含める）

## 2. 生成スタック（訴訟リスク最小化の3層構造）

原則: **下の層で足りるなら上の層を使わない。**

### 第1層（デフォルト）: プロシージャル生成 — 依拠性ゼロ・著作権主張可

| 領域 | 技術 | ライセンス |
|---|---|---|
| 環境音（雨・風・波・焚き火・ノイズ） | **Elementary Audio + node-web-audio-api**（TSネイティブ、offline-renderer）。設計はFarnell『Designing Sound』を自前移植。品質不足時はSuperCollider NRT | MIT / MIT（GPLはサブプロセスなら無関係。出力音声にツールライセンスは及ばない） |
| 映像（シェーダ・パーティクル・フローフィールド・シーンCG） | **three.js/p5.jsヘッドレスレンダリング**（Puppeteerフレームキャプチャ→ffmpeg）＋ffmpeg内蔵ジェネレータ・zoompan・gl-transitions | MIT |
| オーバーレイ（タイマー・曲名・進行表示） | ffmpeg drawtext / three.jsに統合 | — |

- コードとパラメータを人間が全て書くため、**生成物に人間の著作物として著作権を主張できる**
  （AI生成物の「著作権不発生→コピーを止められない」問題を回避）。
- seed駆動の決定論的レンダリングにし、`uniqueness.json`（§6.1）にパラメータを記録。

### 第2層（品質が必要な箇所のみ）: オープンウェイトAIモデル — Apache 2.0/MITのみ

| 領域 | モデル | 条件 |
|---|---|---|
| BGM（メロディのある音楽） | **Stable Audio Open 3.0 Small/Medium**（訓練データがライセンス済みと公式言明・6分超生成・年商$1M未満商用可） | Stabilityへの商用登録。年商$1M監視 |
| BGM補完 | ACE-Step 1.5（Apache 2.0、Mac/MLX対応） | 訓練データ非開示のため類似チェック必須 |
| 静止画（背景・キーアート） | FLUX.1 schnell / FLUX.2 klein **4B** / Qwen-Image / Z-Image Turbo（全てApache 2.0） | HF公式リポジトリのみ。**Civitai派生・マージ済みチェックポイントは全面禁止** |
| 動画AI（当面不使用） | 必要になったらWan 2.2（Apache 2.0）をA2000 12GBかクラウドで | Macでは非実用的（2秒に80分） |

- AI音楽を使う動画は**開示トグル必ずON**（公式例示「AI generated music」。ラベル自体の不利益は公式に否定）。
- 映像スタイルは**非フォトリアル（アニメ調・抽象・様式化）に固定**: 開示論点とプレーヤー上ラベルを回避し、
  実在風景との類似性も構造的に生じにくい。
- BGMは生成のみで終わらせず、**人間の編曲・構成・ミックス工程を挟む**（著作物性の確保＋文化庁チェック
  リスト整合）。作業ログを保存。

### 第3層（原則不使用）: 商用生成サービス

- Suno: フェーズ0の品質ベンチマーク比較のみ（有料プラン・即DL・規約スナップショット保存）。
  UMG/Sony訴訟が2027年本案のため主軸にしない。Udioは外部利用不可となり不採用。
- MusicGen（非商用）・Tencent系（学術限定）・FLUX dev系の商用運用（グレー）・SD3.5/SVD（年商制限）は不使用。

## 3. ニッチ選定 — データで決める（フェーズ0の最初のタスク）

調査4章の結論から候補は3つ。**既存の research パッケージで定量検証してから1つに絞る。**

| 候補 | 強み | 収益構造 |
|---|---|---|
| **A. TRPG/ファンタジー環境音**（酒場・ダンジョン・雨の宿屋・冒険者ギルド） | **日本語圏ほぼ空白**なのに国内TRPG人口大。シーン定義がプロシージャルのパラメータ設計と一致。「セッション用」という機能性がinauthentic判定への防御 | 広告＋**GM向け素材ライセンス販売・メンバーシップ**（Ghelfiモデル実証済み・客単価高） |
| **B. ポモドーロ/集中タイマー×世界観アンビエンス** | タイマーUIが**完全プロシージャル＝毎本の実質差分**を機械的に作れる。検索起点（"勉強タイマー 50/10"）で老舗と競合しない。環境系新チャンネルが6ヶ月で10万登録の実証例あり | 広告＋ライブ「作業ルーム」＋メンバーシップ |
| **C. 物語性サウンドスケープ×和/乗り物**（夜行列車・雨の和室・囲炉裏） | 睡眠系の高RPMを取れる唯一の低飽和ルート。「和」は英語圏への輸出資産。既存構想と地続き | 広告＋Spotify配信 |

**検証手順**（追加実装ほぼゼロ）:
1. `research-data/keywords.yaml` に3ニッチ分のキーワードセットを追加
   （例: A「TRPG BGM 酒場」「dnd tavern ambience」、B「勉強タイマー」「pomodoro 50 10」、
   C「夜行列車 音」「rain japanese room」）
2. `research discover` → `candidates` で各ニッチの競合を台帳化、`collect` を2〜4週間回す
3. エッジ指標で「**伸びているのに投稿頻度・品質が低い**」谷が最も深いニッチを主軸に採用
   （判断は人間。Webビューで比較）

**設計上の重要な点**: 3候補とも中身は「**シーン特化アンビエンス**」であり、
生成エンジン（§2）・配信基盤（§5）・パイプライン（§4)は共通。
ニッチはその上のパッケージング（ブランド・タイトル・オーバーレイ）の差でしかない。
→ エンジンを共通実装し、ニッチはブランド差し替えで転換・並行できる構造にする。

```
[0] リサーチ/テーマ選定    … research pkg のエッジ指標＋キーワード → theme.json（人間が選択）
        ▼
[1] シーン設計             … LLM: scene.json（世界観・短い物語(Storyline)・音レイヤー構成・
        │                     映像パラメータ・尺）＋ uniqueness.json（固有性マニフェスト）
        ▼
[2] 音響生成               … packages/audio-synth: 環境音プロシージャル合成（チャンク→concat）
        │                     ＋（必要時のみ）Stable Audio 3.0 でBGM → 人間の編曲・ミックス工程
        │  → ambient.wav / bgm.wav / mix.wav ＋ 生成ログ・ステム保存
        ▼
[3] 映像生成               … packages/visual-synth: three.jsヘッドレス or 静止画＋ffmpeg微動
        │  → visual.mp4（完全ループ・非フォトリアル様式）
        ▼
[4] 合成                   … ffmpeg: 映像×音声×オーバーレイ（タイマー/曲名等）→ final.mp4
        │                     ＋配信用セグメント（§5の共通エンコードプロファイル）
        ▼
[5] メタデータ生成          … LLM: タイトル(SEO語彙はresearchから)・説明(Storyline＋チャプター)・
        │                     タグ・サムネ3案。効能表現リンターで機械チェック
        ▼
[6] 人間レビュー（必須）    … 品質・権利（逆画像検索/ACRCloud照合）・固有性・開示設定の確認
        ▼
[7] 投稿                   … 限定公開でContent IDスキャン確認 → publishAtで公開予約・
        │                     再生リスト追加・AI開示ON（プレミア公開は手動）
        ▼
[8] ライブ編成             … packages/live: VPSから定時ブロック配信（§5）
        ▼
```

**`uniqueness.json`・生成ログ・ステム・編集履歴・規約スナップショット**を必須成果物に追加（§6）。

## 5. 配信基盤（packages/live）— ホストはローカルサーバー、必要ならVPSへ移設

### 5.1 設計原則

- **配信ホストは再エンコードしない**。レンダリング済みのセグメントを `-c copy` でRTMP push
  するだけ（CPUほぼゼロ・GPU不要）。ホストはまず**ローカルサーバー（Proxmox上のVM/CT）**、
  自宅回線の上り帯域・安定性が不足する場合のみVPSへ移設（同じDockerイメージで動く形にする）。
- **12時間未満のブロックに区切る**（調査1.3: 12時間超はアーカイブされない。ブロック化で
  アーカイブ二重活用＋4,000時間計上を確実にする）。
- ループ配信のinauthentic対策として**編成をコードで多様化**: プレイリストローテーション・
  時刻連動シーン（夜→深夜→夜明け）・動的オーバーレイ。

### 5.2 アーキテクチャ

```
[レンダリング] ローカルサーバー（CPU/プロシージャル）＋ LAN内GPUノード（ComfyUI等をHTTP API化）
   └─ segments/<scene>/<seg>.mp4（共通プロファイル: 1080p30 H.264 高profile,
      ~4-6Mbps, keyint固定, AAC 160k/48kHz → concatで無劣化連結可能）
        │ rsync（VPS移設時のみ。ローカル配信ならファイル共有で直結）
        ▼
[配信ホスト] ローカルサーバー → 必要ならVPSへ移設
   packages/live デーモン（Docker + systemd, Restart=always）
   ├─ scheduler: schedule.yaml（ブロック定義: 開始/終了時刻・シーン・プレイリスト規則）
   │     → ブロックごとに playlist.ffconcat を生成
   ├─ broadcaster: YouTube API liveBroadcasts.insert（タイトル・開始時刻）
   │     → liveBroadcasts.bind（永続streamに紐付け）→ transition(live/complete)
   ├─ pusher: ffmpeg -re -f concat -safe 0 -i playlist.ffconcat -c copy -f flv rtmp://…
   └─ watchdog: liveStreams.list の status/healthStatus 監視・ffmpeg自動再起動・通知
```

### 5.3 段階実装（§7のフェーズ3〜4に対応）

| Step | フェーズ | 内容 | 実装量 |
|---|---|---|---|
| **0** | 3 | ライブ有効化（電話確認・待機期間）。**配信はローカルサーバーから**（systemd timer＋ffmpeg、夜間の定時ブロック<12h、手動起動で開始してよい） | 手作業＋極小 |
| **1** | 3 | 定時配信の安定化: systemd自動化、「デフォルト配信」永続キー、アーカイブ公開運用。編成ルール・視聴データ・**自宅回線の上り実効帯域**の観察 | 極小 |
| **2** | 4 | **配信ホストの確定**: フェーズ3実測でローカル常設 or VPS移設を判断。移設時はrsync＋systemd一式をVPSへ | 小 |
| **3** | 4 | broadcaster実装: APIで放送作成・バインド・遷移・タイトル自動化。ブロック<12hの自動分割。再生リスト追加 | 小 |
| **4** | 4 | scheduler＋時刻連動シーン切替、watchdog（ヘルス監視・自動復旧・失敗通知）、Docker化（ローカル/VPSどちらでも同イメージ）。24/7化の判断 | 中 |
| **5** | 4 | YPP通過後: `liveBroadcasts.cuepoint`/monetizationDetailsで広告制御 | 小 |

### 5.4 帯域見積り（Step 0で実測確認すること）

- 1080p30 6Mbps: 約65GB/日 ≒ **2TB/月**（24/7時）。毎晩9時間なら約0.7TB/月。
- アンビエント映像は動きが少なく圧縮が効くため、**720p/3Mbps（約1TB/月@24/7）への引き下げも許容範囲**。
  VPSの転送量上限・上り実効速度と照合して解像度/ビットレートを決める。

### 5.5 リスク対応

| リスク | 対策 |
|---|---|
| ffmpeg/回線断 | systemd Restart=always＋watchdog。切断猶予は非公開仕様のため、再接続は数十秒以内を目標に |
| キーのローテーション | watchdogがstream状態を検知して通知。キーは環境変数/秘匿ファイルで管理 |
| 虚偽DMCA/誤クレームで配信停止（Lofi Girl事件型） | 全素材の証跡（§6）を即提示できる状態に。配信はブロック化しているため単一停止の影響が限定的 |
| 同一素材の使い回し判定 | 編成規則で「同一セグメント列の再放送」を禁止（schedulerがシャッフル＋日次で新規セグメント混入） |

## 6. コンプライアンス・プレイブック

### 6.1 固有性の機械的保証（inauthentic対策の本丸）

- `uniqueness.json`（動画ごと必須成果物）: シーンseed・音レイヤー構成・BGMトラックID・
  映像パラメータ・オーバーレイ差分を記録し、**過去動画との重複をCIで機械チェック**
  （同一seed・同一トラック列の再利用をビルド失敗にする）。
- 各動画に**Storyline（シーン設定の短文）**を説明欄に付与（New Bliss型・人間キュレーションの証跡）。
- チャンネルは**自社IP（キャラクター/世界観）**を最初から設計（Lofi Girl型防御）。
- 投稿頻度は常識的範囲（週2〜3本目安）。**YPP申請は差別化された動画が揃ってから**。
- **メタデータも非テンプレ化**: タイトルの連番テンプレ（"〇〇 Mix #47"型）・同一構造の説明文の
  連発を避ける（審査シグナルとして指摘されている）。
- **制作解説動画を混ぜる**: 声なしチャンネルでも「制作の裏側」動画（パイプラインの画面収録等）を
  時々挟み、人間の創作関与を可視化する（審査官は人気動画を重点チェックするため、
  代表作に独自性を強く入れる）。
- ⚠️ 一次事例の教訓（[環境音チャンネル収益化実態](../reference/2026-07-16-調査全文-環境音チャンネル収益化実態.md)）:
  「AI静止画＋AI音楽＋軽エフェクト」構成のチャンネルは**制作証明を提出しても再審査却下された例がある**。
  確実な方法は存在しない前提で、映像の動き・構成差分・機能性に投資する。
  却下時の定石は「問題動画の非公開化→明確にオリジナルな動画を5〜10本追加→30日（2回目以降90日）
  待って再申請」。
- 保険: アグリゲーター経由のArt Tracks（YouTube Music）はチャンネルのYPP審査なしで
  音源収益化できる代替経路として存在する（自社曲に著作物性を確保できている場合の選択肢）。

### 6.2 開示・表現

- **AI開示トグルは常にON**（AI音楽使用時は公式例示上必須。非フォトhad映像なら説明欄内ラベルのみ）。
- 映像は非フォトリアル様式に固定。実在地名を冠したフォトリアル映像は作らない。
- 効能表現リンター: 「治る」「改善」「〇〇Hzで…」等のNG辞書でタイトル・説明を機械チェック
  （声なしでも睡眠・集中を謳う場合のメタデータには適用される）。

### 6.3 証跡主義（権利防衛）

各 `assets/<video_id>/` に保存: 生成ログ（プロンプト・seed・モデル/ツールのバージョン）、
中間ステム、人間の編集作業ログ、使用ツールの規約スナップショット、公開前タイムスタンプ。
→ Content ID誤クレームのdispute・YPP再審査・依拠性否認の三役。
自作曲のContent ID登録は**しない**（ディストリビューターがAI曲登録を拒否傾向・誤クレームの温床）。

### 6.4 公開前ゲート（人間レビュー[6]に統合）

1. 逆画像検索（映像キーフレーム）
2. ACRCloud照合（メロディのあるBGMのみ）
3. **限定公開アップロード→Content IDクレーム有無を確認→公開予約**
4. 開示設定・効能リンター結果・uniqueness重複チェックの確認

## 7. フェーズ計画（VPSは最終盤。小さく登る）

各フェーズに**出口条件**を置き、満たすまで次へ進まない（YAGNI維持）。
インフラは フェーズ0〜3はMac完結、**VPS投入はフェーズ4のみ**。

### フェーズ0 — ニッチ検証（Mac・手動中心・作り込まない）

やること:
1. **research**: keywords.yamlに3ニッチ（TRPG環境音/ポモドーロ/和サウンドスケープ）の
   キーワード追加、discover/collect を2〜4週間運用（実装済み機能の運用のみ）
2. **packages/audio-synth 最小版**: Elementaryベースの環境音合成（まず雨1プリセット、
   チャンクレンダリング→ffmpeg concat、seed駆動）
3. **packages/visual-synth 最小版**: three.jsヘッドレス1シーン（パーティクル＋グラデーション
   完全ループ）＋ffmpeg合成
4. **効能表現リンター＋uniqueness.json（軽量版）**（実装コスト極小）
5. 試作動画を**3〜5本、半手動で制作・手動投稿**（候補ニッチの試作を混ぜる）
