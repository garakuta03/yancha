## 1. 概要 / ゴール

睡眠導入系のAI動画をYouTubeに投稿し、収益化（YouTubeパートナープログラム）を達成する。
手作業の動画制作ではなく、**完全オリジナル・自社ライセンスのアセットを再現生産する自動化パイプライン**を
エンジニアリングで構築し、資産として積み上げる。

### 成功指標
- **フェーズ0**: 収益化審査を通過し得るフォーマットを実データで1つ特定する（勝ち筋の発見）。
- **収益化条件（参考）**: チャンネル登録者 1,000人 ＋ 直近12ヶ月の総再生時間 4,000時間（またはShortsの規定値）。
  - 睡眠系は長尺・リピート視聴で総再生時間を稼ぎやすい。
- **フェーズ1**: 勝ちフォーマットに特化したパイプラインで、品質を保ったまま投稿を継続生産できる状態。

## 2. 設計の大前提（不変の原則）

> **ホワイト運用 × 完全オリジナル・自社ライセンス。**
> 「飽和 × 量」ではなく「防御可能なニッチ × 完全オリジナルのアセット」で戦う。
> 量産はするが、それは"薄いコンテンツの数"ではなく"独自フォーマットの再現生産"である。

- 全アセット（台本・音声・音楽・映像）は自分がライセンスを保有する状態にする。
- 各動画は変換的・独自であり、「再利用コンテンツ」判定を回避する。
- 断定的な健康効能を謳わない。

## 3. コンテンツフォーマット

| 案 | 内容 | リスク | 位置づけ |
|---|---|---|---|
| **①（主軸）** | オリジナル睡眠導入ガイド朗読（呼吸誘導・ボディスキャン・情景描写） | ★☆☆ | フェーズ0の1本目・検証の中心 |

## 4. パイプライン・アーキテクチャ

各動画は独立して理解・テスト可能なステージの直列で構成する。ステージ間はファイル成果物と
メタデータ（JSON）で受け渡し、どのステージからでも再実行できるようにする。

```
[0] 需要リサーチ / テーマ選定   … LLM(ChatGPT/Gemini) + 手動キュレーション
        │  → theme.json（テーマ・キーワード・想定尺）
        ▼
[1] 台本生成                    … LLM。オリジナル台本（呼吸誘導/情景）
        │  → script.md（＋話速・間の指示）
        ▼
[2] ナレーション音声生成        … TTS（にじボイス / ElevenLabs 等・商用ライセンス）
        │  → narration.wav
        ▼
[3] 音楽・環境音生成            … AI音楽（商用ライセンス）or ローカル生成
        │  → bgm.wav / ambient.wav
        ▼
[4] 音声ミックス                … ナレーション＋BGM＋環境音（ffmpeg / 音量・間の調整）
        │  → mix.wav
        ▼
[5] 映像生成                    … ローカルA2000（ComfyUI + SDXL、微動画はSVD/AnimateDiff）
        │  → visual.mp4（ループ素材）
        ▼
[6] 動画合成                    … ffmpeg で映像×音声を尺に合わせて合成
        │  → final.mp4
        ▼
[7] メタデータ生成              … LLM（タイトル・説明・タグ・サムネ案）
        │  → metadata.json
        ▼
[8] 人間レビュー（必須の1段）    … 品質・権利・効能表現チェック（スパム/自動化ポリシー回避）
        ▼
[9] 投稿                        … YouTube Data API v3（初期は手動UPでも可）
```

### 設計上の原則
- 各ステージはCLI/関数として単体実行・単体テスト可能にする。
- 成果物は動画IDごとのディレクトリに集約（`assets/<video_id>/...`）。
- **ライセンス証跡**（使用ツール・プラン・利用規約バージョン）を `license.json` として各動画に保存。
- **[8]の人間レビューは自動化しない**（スパム/自動化ポリシー・品質担保・効能表現の最終防波堤）。

## 4.1 技術スタック方針

- **言語: TypeScript 基本。スクリプト/ステージは tsx で実行。**
  - オーケストレーション・各ステージのランナー・YouTube APIクライアント・LLM/TTS/音楽APIクライアントはTSで実装。
  - ComfyUI（Python製）は自前実装せず、**HTTP API越しにTSから呼び出す**。ffmpegはCLIをTSから起動。
- **Docker化: できるものはコンテナ化する（再現性・サーバー移行の容易さ）。**
  - オーケストレーション、ffmpeg合成、各ステージランナーはコンテナ化対象。
  - ⚠️ ローカルGPU（A2000/Windows）のComfyUIはGPUパススルー（WSL2 + NVIDIA Container Toolkit）が必要。
    初動はホスト直起動でも可、将来のLinuxサーバーでコンテナ化を本格化。

## 5. インフラ構想

| フェーズ | 生成(GPU) | オーケストレーション | 備考 |
|---|---|---|---|
| 初動 | A2000（Windows, ローカル） | Mac（MBA/MBP） | ComfyUIをAPIサーバー化し、MacからHTTPで叩く |
| 将来 | A2000（外部GPUノード） | Linuxサーバー（主） | サーバーがジョブを管理し、GPUノードをリモート呼び出し |

- ComfyUIをAPIモードで常駐させることで、初動→将来のサーバー移行がスムーズ。
- ⚠️ A2000のVRAM（6GB/12GB）を要確認。SDXLは実質8〜12GB必要。動画生成（SVD/AnimateDiff）はさらにVRAMを要求するため、
  解像度・フレーム数の最適化、またはM2 Max（ユニファイドメモリ）での代替を検証すること。

## 6. ツール候補（フェーズ0で確定する）

| 役割 | 第一候補 | 備考 / 確認事項 |
|---|---|---|
| 台本・メタデータ | ChatGPT / Gemini（契約済） | 追加コストなし |
| ナレーション(TTS) | にじボイス（日本語自然）/ ElevenLabs | 睡眠向けの落ち着いた声。**商用・収益化ライセンスを要確認** |
| 音楽・環境音 | Suno（Pro/商用）/ Udio / ローカル生成 | **無料プランは商用NGが多い。収益化明記プランのみ採用** |
| 画像・映像 | ローカル ComfyUI + SDXL、微動画 SVD/AnimateDiff | チェックポイントの**商用ライセンスを記録** |
| 動画合成 | ffmpeg | ローカル無料 |
| 投稿 | YouTube Data API v3 | 初期は手動UPでも可。APIクォータ確認 |

## 7. フェーズ計画

### フェーズ0 — 検証（YAGNI: 作り込まない）
- 目的: 「どのフォーマットが再生され、収益化審査を通るか」を実データで確認し、勝ち筋を1つ特定。
- 進め方: 案①を中心に **5〜10本**を手動〜半自動で制作・投稿。案②も後半に単体投入して審査挙動を観察。
- 完了条件: 収益化を狙えるフォーマットが特定でき、各ステージの手順が言語化されている。

### フェーズ1 — 自動化
- 目的: 勝ちフォーマットに特化したパイプラインを段階的に構築。
- 範囲: 上記アーキテクチャの各ステージを、勝ち筋に絞って自動化。人間レビュー[8]は残す。
