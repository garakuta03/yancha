# 設計ドキュメント — リサーチ発掘（プランB＋C）

- 日付: 2026-07-16
- ステータス: 承認済み（実装計画待ち）
- 対応Issue: #16（プランB=キーワード自動発掘）/ #18（プランC=手動＋自動補充ハイブリッド運用）
- 前提: [プランA設計](2026-07-16-リサーチ定点観測サブシステム-design.md)（`@yancha/research` の台帳/collect/store/report/serve）の上に構築する。

---

## 1. 目的 / ゴール

プランA（手動シード台帳＋定点観測）に、**キーワードによる競合チャンネル自動発掘（B）**と、
**手動シードと発掘分を統合運用するポリシー（C）**を足す。監視対象を人手だけに頼らず広げつつ、
台帳の健全性（ノイズ・死んだ対象の排除）を保つ。

### スコープの方針
- BとCは密結合のため**1スペック・1実装計画**で扱う。B＝発掘メカニズム、C＝統合運用ルール。
- フェーズ0の原則を踏襲：**Mac完結・常駐なし・コマンド駆動・作り込まない（YAGNI）・判定は人間**。

## 2. 不変原則との整合

- **判定は人間**：発掘した候補は台帳へ自動投入しない。`candidates.yaml` に貯め、人間が承認したものだけ台帳へ昇格する。
- **常駐なし・コマンド駆動**：`discover` はcron化せず、気が向いたとき手動実行する。「定期」はコマンドの手動実行で担保。
- **クォータ保護**：`search.list`（100ユニット/回）の誤爆で枠を溶かさないよう、実行前に見積り・ガードする。
- **スクレイピング禁止**：公式 YouTube Data API v3 のみ。

## 3. YouTube Data API のコスト設計

- 発掘は `search.list`（`part=snippet`, `type=channel`）＝**1回100ユニット固定**。1キーワード＝1回（ページングしない）。
- 発掘で得るのは channelId とスニペットのみ。**統計（登録者数等）はプランA既存の `channels.list`（1ユニット系）で肉付け**する。
  肉付けは channelId を最大50件ずつまとめて叩く（`ceil(件数/50)` ユニット）。
- 1回の `discover` 見積り: `キーワード数 × 100 ＋ ceil(取得channelId数 / 50)`。
  例: 10キーワード・重複除外後120件 → 1000 ＋ 3 = 1003ユニット（10,000/日に対し余裕）。

## 4. アーキテクチャ（プランAへの追加）

新規ステージ・コマンドを `@yancha/research` に追加する。プランAの `youtube`/`store`/`ledger` を再利用する。

| # | 名前 | 役割 | 入出力 |
|---|---|---|---|
| B1 | キーワード台帳 | 発掘の入力を手動管理 | `keywords.yaml`（keywords[]＋発掘設定） |
| B2 | 発掘（discover） | 各キーワードで `search.list` → 統計肉付け → フィルタ → 候補保存 | keywords.yaml → `candidates.yaml` |
| C1 | 候補レビュー | 候補の承認/却下 | `approve`/`reject <channelId>` → ledger.yaml / candidates.yaml 更新 |
| C2 | 退役（retire） | 死んだ/不要チャンネルを監視対象から外す | `retire <channelId>` → ledger.yaml（`retiredAt` 付与） |

### コマンド（プランAに追加）
```bash
pnpm research discover [--yes]      # keywords.yaml で発掘 → candidates.yaml に候補を貯める
pnpm research candidates            # 未承認の候補一覧を表示
pnpm research approve <channelId>   # 候補を台帳へ昇格（source=discovered, approvedAt付与）
pnpm research reject <channelId>    # 候補を candidates.yaml から削除
pnpm research retire <channelId>    # 台帳エントリを退役（collect対象から除外）
```

### データフロー
```
keywords.yaml ─▶ [discover: search.list → channels.list → フィルタ] ─▶ candidates.yaml
                                                                          │ (人間レビュー)
                                                          approve ────────┼──▶ ledger.yaml (source=discovered)
                                                          reject  ────────┘     │
                                              retire ───────────────────────────▶ ledger.yaml (retiredAt)
                                                                                  │
ledger.yaml(退役除外) ─▶ [collect(プランA)] ─▶ ...（以降プランA）
```

## 5. フィルタ設計（プランB・候補のノイズ削り）

`discover` は候補化の前に機械フィルタをかける（人間の承認負荷を実用域に落とす）。

1. **重複排除**: 既に `ledger.yaml`（退役含む）または `candidates.yaml` に存在する channelId を除外。
2. **登録者数の下限**: `minSubscribers` 未満を除外（既定 100）。
3. **言語/地域**: `search.list` に `relevanceLanguage`（既定 `ja`）・`regionCode`（既定 `JP`）を付けて日本語圏に寄せる。
4. **件数上限**: 1回の `discover` で候補化する上限 `maxCandidatesPerRun`（既定 30）。登録者数の多い順に採用。

閾値は `keywords.yaml` の設定セクションに持たせ、後から調整可能にする。

## 6. データ設計

### `keywords.yaml`（人間が手編集）
```yaml
keywords:
  - 睡眠導入
  - ヒーリング 音楽
  - ボディスキャン 瞑想
settings:
  minSubscribers: 100
  maxCandidatesPerRun: 30
  relevanceLanguage: ja
  regionCode: JP
  quotaGuardUnits: 3000   # 見積りがこれを超えたら中断（--yesで続行）
```

### `candidates.yaml`（discover が生成・人間がレビュー）
```yaml
candidates:
  - channelId: UCxxxx
    title: "眠れるチャンネル"
    subscriberCount: 12000
    matchedKeywords: ["睡眠導入"]
    discoveredAt: 2026-07-16T09:00:00.000Z
```

### `ledger.yaml` の拡張（プランAの `LedgerEntry` に追記）
- 追加フィールド:
  - `source: "manual" | "discovered"`（既存の手動追加は `manual`。未設定エントリは `manual` とみなす＝後方互換）
  - `approvedAt?: string`（`discovered` を承認した時刻）
  - `retiredAt?: string`（退役時刻。存在すれば collect 対象から除外）
- 既存 `add`（手動シード）は `source: "manual"` を付与する。
- `collect` は `retiredAt` を持つエントリを除外して回す（プランA `collect` の入力段でフィルタ）。

### 出自タグの意義（プランC・A）
`source` と `approvedAt` により、後で「発掘経由の監視対象が実際に良い競合だったか」を効果検証できる。

## 7. 退役（プランC・B）

- `research retire <channelId>` は台帳エントリに `retiredAt` を付与するだけ（行は消さない＝履歴保持）。
- **自動退役はしない**（死亡判定の自動化はスパム/自動化ポリシー観点でも避け、人間が判断）。
- 退役済みは `collect` の対象から外れ、既存スナップショット（時系列）は残る。

## 8. クォータガード（プランC・C）

- `discover` は実行前に見積りユニット（`キーワード数 × 100` の search 分）を算出し表示する。
- 見積りが `settings.quotaGuardUnits` を超える場合、`--yes` が無ければ中断してユーザーに確認を促す。
- 肉付けの `channels.list` 分（1ユニット系）はガード対象に含めず表示のみ（コストが小さいため）。

## 9. 技術方針

- 既存 `@yancha/research` に追加実装（新パッケージは作らない）。
- `youtube.ts` に `searchChannels(keyword, opts)`（`search.list`）を追加。統計肉付けは既存 `fetchChannels` を再利用。
- YAML 入出力は既存 `ledger.ts` と同じ `yaml` パッケージ。候補/キーワードも同様。
- テストは vitest。`search.list` 呼び出しは `fetchImpl` 差し替えでモックする（プランA `youtube.test.ts` と同方式）。
- Webビューへの候補表示は本スコープ外（#17 ダッシュボードで扱う）。フェーズ0はCLIで承認する。

## 10. スコープ / 非スコープ

### 本設計の範囲
- キーワード台帳・discover（search＋肉付け＋フィルタ）・candidates・approve/reject・retire・出自タグ・クォータガード。

### 非スコープ（別Issue）
- 候補のWeb表示・承認UI → #17（ダッシュボード）。
- キーワードの自動生成（台帳タグから）・ページング深掘り → 勝ち筋が見えてから（YAGNI）。
- 未承認候補の自動失効（棚卸し）→ 候補が溜まる運用になってから。
