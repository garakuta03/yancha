## 1. 目的 / ゴール

フェーズ0の目標は「勝ち筋（収益化審査を通り、再生されるフォーマット）を1つ特定する」こと。
そのための**エッジ探し（調査系）を仕組み化**するサブシステムを構築する。

具体的には、手動でシードした競合チャンネルを**コマンド駆動でスナップショット取得** →
時系列で蓄積 → **フォーマットの隙間（エッジ）をWebビューで提示**する。判定は人間が行う。

将来的には、生成パイプラインの進捗・投稿実績・ライセンス証跡も同じWebアプリに載せ、
**運用総合ダッシュボード**へ育てる。本サブシステムはその土台となる。

### 成功指標（フェーズ0）
- 手動シードした競合を、常時起動なしで定点観測できる。
- 尺・投稿頻度・伸び率・タイトル/タグ傾向を1画面で比較でき、参入余地の当たりがつく。

## 2. 設計の大前提（本プロジェクトの不変原則との整合）

- **ホワイト運用**: スクレイピングは使わない。公式 YouTube Data API v3 のみ。
- **Mac完結・常駐なし**: cron/デーモン/Docker常駐を作らない。すべて**オンデマンドのコマンド駆動**。
- **判定は人間**: `report`/Webビューはランキング・異常値・空白ゾーンを"提示"するだけ。勝ち筋を断定しない。
- **YAGNI**: フェーズ0では手動シード（プランA）のみ。自動発掘（B/C）は後続で別ステージとして足す。
- **将来移行の容易さ**: 同じコマンドを将来Linuxサーバーのcronで叩くだけに落ちる作りにする。

## 3. リポジトリ構成

- **同一リポジトリ内のパッケージ分割**（monorepo）。調査系を別リポには分けない。
- 配置: `packages/research/`（独立 workspace パッケージ `@yancha/research`）。
- **共有モジュールは `packages/core`（`@yancha/core`）へ昇格**して両パッケージから使う。
  対象は汎用部のみ（`YanchaError`/`Logger`/`io/json`）。生成パイプライン固有の `config`（LLMプロバイダ等）は
  昇格せず、`research` は自前の `config`（`YOUTUBE_API_KEY` 等）を持つ。既存の root パッケージ（生成パイプライン）も
  共有部の import を `@yancha/core` に張り替える。
- 根拠: 調査系の出力（テーマ・キーワード・想定尺）は生成パイプラインの stage[0]「需要リサーチ／
  テーマ選定」の入力になり、同じ語彙（`theme.json` 等）を共有する。Issue台帳も1本（`garakuta03/yancha`）に保てる。
  外部公開・OSS化・別マシン常駐の予定は現時点で無いため別リポには分けない。関心は workspace パッケージ境界で分離する。

## 4. YouTube Data API v3 のコスト方針

- 金銭課金は無い。制約は**クォータ（デフォルト 10,000 ユニット/日）**。
- `search.list` は 1 回 **100 ユニット**と高コスト。**プランAでは search.list を一切使わない。**
- 定点追跡は 1 ユニット系のみで構成する:
  - `channels.list`（統計）= 1
  - `playlistItems.list`（アップロード一覧）= 1
  - `videos.list`（動画統計・尺・タグ）= 1
- 目安: 100 チャンネル監視でも 1 回の `collect` で約 300 ユニット。10,000/日に対し十分な余裕。
- 自動発掘（プランB）で `search.list` を導入する際は、頻度を落として（週1等）別ステージ化する。

## 5. アーキテクチャ

各ステージは tsx で単体実行・単体テスト可能にする（本プロジェクトのパイプライン原則を踏襲）。

| # | 名前 | 役割 | 入出力 | API消費 |
|---|---|---|---|---|
| 1 | 台帳（ledger） | 監視対象チャンネルを手動管理 | `packages/research/ledger.yaml`（channelId・メモ・タグ） | 0 |
| 2 | 収集（collect） | 台帳を回して「今」のスナップショットを取得 | ledger → `snapshots/<ISO8601>/*.json` | 1系×3/ch |
| 3 | 蓄積（store） | スナップショットを時系列DBに追記 | snapshots → SQLite（`data.db`） | 0 |
| 4 | 分析（report） | 指標を集計しWebビューに供給 | DB → Webアプリ（`serve`） | 0 |
| 5 | Webビュー（serve） | オンデマンドのローカルWebアプリで可視化 | DB直読み → `localhost:<port>` | 0 |

### データフロー
```
ledger.yaml ──> [collect] ──> snapshots/ ──> [store] ──> data.db ──> [report/serve] ──> localhost
   (人間)      (手動コマンド)   (生JSON保存)   (時系列)               (見たい時に起動)
```

### コマンド（コマンド駆動・常駐なし）
```bash
pnpm research add <channel_url|id>   # 台帳にチャンネルを追加（手動シード）
pnpm research collect                # 台帳を回して「今」のスナップショットを撮る
pnpm research serve                  # localhost に Webビューを起動（見たい時だけ）
```
- `collect` は `store`（DB追記）まで一括で行う。スナップショット生JSONも `snapshots/` に残す。
- cron・常駐・Dockerはフェーズ0では作らない。将来サーバー移行時は同じ `collect` を cron で叩くだけ。

## 6. データ設計

### スナップショット（生JSON）
- 保存先: `packages/research/snapshots/<取得ISO8601タイムスタンプ>/<channelId>.json`
- API 応答をほぼ生で保存する（後から新しい指標を追加再計算できるようにするため）。
- **各スナップショットに実測タイムスタンプを必ず記録する。「日次」を前提にしない。**

### ストレージ抽象化（SQLite → 将来MariaDB）
- **一旦 SQLite に保存**。ファイル1個で完結し Mac完結の原則に合う。
- 将来 MariaDB 等へ移行する可能性があるため、**storage層をリポジトリ・インターフェースで薄く抽象化**する。
  `collect`/`report`/`serve` はインターフェース経由でのみDBに触り、実装swap時に上位を触らずに済む形にする。
  （SQLite/MySQL 両対応のクエリビルダ〈Kysely 等〉採用も可。実装計画で確定。）

### 主なテーブル（時系列）
- `channels`: channelId, タイトル, 登録者数, 総再生数, 取得タイムスタンプ（スナップショット単位で追記）
- `videos`: videoId, channelId, タイトル, 公開日, 尺, 再生数, タグ, 取得タイムスタンプ（同上）
- 追記型（更新ではなく行を積む）にし、時系列で差分を取れるようにする。

## 7. 分析（エッジ指標）

`report`/Webビューが提示する指標。**伸び率はスナップショット間の実経過時間で正規化する**
（例: `views差 ÷ 実経過時間`）。取得間隔がバラバラ・週1・2日空きでも指標が壊れない。

- **再生速度**（views / 経過日数）: 今伸びている題材
- **投稿頻度 × 登録者伸び**: 割に合うフォーマット
- **尺の分布**: 睡眠系の勝ち尺（1h / 3h / 8h 等）
- **タイトル/タグの頻出語 vs 空白**: 参入余地

## 8. Webビュー

- **オンデマンドのローカルWebアプリ**（`serve` で起動、常駐しない）。
- 構成イメージ: 軽量サーバー（Hono 等）＋ SQLite 直読み ＋ 1ページのチャート。
- フェーズ0の画面は「監視チャンネル一覧 ＋ 伸び率 ＋ 尺分布」の1画面のみ（作り込まない）。
- チャートの配色・設計は実装時に dataviz スキルで統一する。
- **将来の運用総合ダッシュボードの土台**とする。生成パイプライン進捗・投稿実績・ライセンス証跡は
  同じアプリにパネル/ページを追加して拡張する（静的HTML方式は採らない＝作り直しを避けるため）。

## 9. 技術スタック

- 言語: TypeScript、実行は tsx（本プロジェクト方針を踏襲）。
- DB: SQLite（`better-sqlite3`。将来 MariaDB 移行を見据え `ResearchStore` インターフェースで storage層を抽象化）。
- Webサーバー: Hono ＋ `@hono/node-server`。JSON API ＋ 静的HTML1枚、チャートは Chart.js（CDN）。
- 台帳: YAML（`yaml` パッケージ）。人間が手編集しやすい形式。
- API クライアント: YouTube Data API v3。Node 20 の global `fetch` を使用（SDK不要）。`search.list` はプランAでは不使用。
- テスト: vitest（既存基盤に合わせ globals 有効）。
- Docker化はフェーズ0では行わない（常駐なしのため）。将来サーバー移行時に検討。

## 10. スコープ / フェーズ

### 本設計の範囲（プランA・フェーズ0）
- 手動シード台帳 → コマンド駆動の collect/store → SQLite蓄積 → オンデマンドWebビュー。
- `search.list` 不使用。判定は人間。常駐なし。

### 後続（別Issue・別設計）
- **プランB**: キーワードで自動発掘（`search.list` を週次・別ステージ）→ 台帳に自動追加。
- **プランC**: A+B ハイブリッド（手動シード＋定期自動補充）。
- **運用総合ダッシュボード化**: 生成パイプライン進捗・投稿実績・ライセンス証跡を同Webアプリへ統合。
