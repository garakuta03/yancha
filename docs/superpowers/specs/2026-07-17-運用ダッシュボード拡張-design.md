# 設計 — 運用総合ダッシュボード拡張（第1弾: 動画一覧＋詳細＋レビュー記録）

- 日付: 2026-07-17
- 対応Issue: #17
- 上位: [リサーチ定点観測サブシステム](2026-07-16-リサーチ定点観測サブシステム-design.md) / [P0 E2E MVP 設計](2026-07-16-p0-e2e-mvp-design.md)
- ⚠️ この design は Codex 実装の唯一の入力を兼ねる（§6 の段階分割をそのまま実装する）

## 1. 目的
既存の `packages/research` の Hono Webアプリを拡張し、生成パイプラインの `assets/<video_id>/` を
read-only で可視化する運用台帳にする。**第1弾は「動画一覧＋詳細＋人間レビュー記録」**。

## 2. 不変原則との整合（受け入れ条件）
| 原則 | 担保 |
|---|---|
| 人間レビューを自動化しない | レビュー完了は**人間がUIのボタンで記録**する。checks等の自動判定は表示のみで、公開判断はしない |
| assets/ のファイル契約を壊さない | **assets/ は read-only**。レビュー状態は research の SQLite に保存（assets/には書かない） |
| 作り直さない | 同じ Hono アプリ・同じ `web/index.html` を拡張（新規SPAにしない） |
| 段階分割できる | §6 の Task 1〜3 |

## 3. アーキテクチャ

### 3.1 assets/ スキャナ（read-only）
- `ResearchConfig` に `assetsDir` を追加（env **`YANCHA_ASSETS_DIR`**、既定 `"assets"`。`resolve` する）。
- 新規 `packages/research/src/videos.ts`:
  - `listVideoSummaries(assetsDir, store): Promise<VideoSummary[]>` — `assetsDir` 直下の各 `<video_id>/` を走査し、
    各メタJSONの**有無**からステージ進捗、`checks.json` の合否、`upload.json` の mode/URL、`store` のレビュー状態を集約。
  - `readVideoDetail(assetsDir, videoId, store): Promise<VideoDetail>` — scene(title/storyline)・checks結果・
    upload(mode/videoId/url)・`review.md` 本文・license entries・reviewedAt をまとめる。
  - パイプラインの成果物は `{ videoId, stageId, createdAt, data:{...} }` でラップされている。**`data` を取り出す純関数**を用意し
    テストする。**欠損ファイルに寛容**（生成途中の動画がある）。
- 型:
  ```typescript
  interface VideoStageProgress { readonly [stage: string]: boolean; } // theme/scene/audio/visual/video/metadata/checks/upload/review
  interface VideoSummary {
    readonly videoId: string;
    readonly stages: VideoStageProgress;
    readonly checksPassed: boolean | null;   // checks.json 未生成なら null
    readonly uploadMode: "dry-run" | "uploaded" | null;
    readonly reviewedAt: string | null;
  }
  interface VideoDetail extends VideoSummary {
    readonly title: string | null;
    readonly storyline: string | null;
    readonly checks: readonly { name: string; passed: boolean; details: readonly string[] }[];
    readonly uploadUrl: string | null;
    readonly reviewMarkdown: string | null;  // review.md の生テキスト
    readonly license: readonly { assetType: string; tool: string; createdAt: string }[];
  }
  ```

### 3.2 レビュー記録（SQLite）
- `store.ts` に **`video_reviews` テーブル**（`video_id TEXT PRIMARY KEY, reviewed_at TEXT NOT NULL, note TEXT`）を追加。
- `ResearchStore` にメソッド追加: `setVideoReviewed(videoId, reviewedAt, note?)` / `getVideoReview(videoId)` / `listVideoReviews()`。
  - **時刻は引数で注入**（`Date.now()`/`new Date()` を store 内で直接使わない。呼び出し側＝server が ISO文字列を渡す）。

### 3.3 API（`server.ts` / Hono）
- `GET /api/videos` → `VideoSummary[]`
- `GET /api/videos/:id` → `VideoDetail`（無ければ404）
- `POST /api/videos/:id/review` → body `{ note?: string }` → `setVideoReviewed(id, <ISO now>, note)` → 200 JSON。**人間操作**。
- 既存 `/api/report` と `/` は維持。

### 3.4 UI（`web/index.html` 拡張・バニラJS）
- 既存リサーチ表の下に **「動画パイプライン」セクション**を追加。`/api/videos` を表で表示
  （video_id・ステージ進捗・checks合否・upload・レビュー済み）。
- 行クリックで `/api/videos/:id` を取得し詳細パネル表示（storyline・checks結果・upload URL・
  `review.md` チェックリスト・license証跡）。詳細に **「レビュー済みにする」ボタン** → `POST /api/videos/:id/review`。
- フレームワークは入れない（現行の素の fetch + DOM 構築に倣う）。

## 4. テスト方針
- `videos.ts` の集約・ラップ外し・欠損寛容・checks合否抽出・upload mode抽出を**純関数として単体テスト**
  （tmpディレクトリにダミー `assets/<id>/*.json` を作って検証）。
- `store` の `video_reviews` CRUD をテスト（既存 store テストに倣う）。
- server は既存作法に倣い最小限（過剰にテストしない）。

## 5. Global Constraints
- TypeScript ESM、import は拡張子 `.js`。コメント/ログ/エラーは日本語。`YanchaError`（code付き）。
- テストは vitest（globals）。`Math.random()`/`new Date()`/`Date.now()` の新規使用禁止（時刻は server で生成し注入）。
- assets/ には**書き込まない**。

## 6. 段階分割（Codex実装タスク）
- **Task 1**: `ResearchConfig.assetsDir` 追加 ＋ `videos.ts`（`listVideoSummaries`/`readVideoDetail`＋`data`取り出し純関数）
  ＋ `GET /api/videos`・`GET /api/videos/:id`。純関数テスト。
- **Task 2**: `store` の `video_reviews` テーブル・メソッド ＋ `POST /api/videos/:id/review`（時刻注入）。CRUDテスト。
- **Task 3**: `web/index.html` 拡張（一覧＋詳細＋「レビュー済みにする」ボタン）。

## 7. 非スコープ（次弾以降）
- 投稿実績の集計パネル（本数・合否率・平均生成時間）。
- YouTube投稿API連携・自動レビュー・自動投稿。
- 初期リサーチWebビューの再実装。
