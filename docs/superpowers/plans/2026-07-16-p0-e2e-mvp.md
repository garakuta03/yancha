# フェーズ0: E2EパイプラインMVP 実装計画

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** `pnpm pipeline --video-id <id>` の一発で、限定公開の完成動画が1本 YouTube に上がり、レビュー用サマリが出る状態を作る。**品質は問わない。形を作ることが目的。**

**Architecture:** 既存の `Orchestrator` ＋ ステージ配列を、朗読前提（`script`/`narration`/`music`/`audioMix`）から声なし前提（`theme → scene → audio → visual → video → metadata → checks → upload → review`）へ組み替える。音・映像の実体は新規パッケージ `@yancha/audio-synth` / `@yancha/visual-synth` に置き、ステージは薄い呼び出し層にする。生成は**プロシージャル第1層のみ**、LLMは scene/metadata のテキストのみ。

**Tech Stack:** 既存（TypeScript ESM / tsx / vitest / pnpm workspace / `@yancha/core`）に追加。ffmpeg は子プロセス。

**設計:** [フェーズ0 E2E MVP 設計](../specs/2026-07-16-p0-e2e-mvp-design.md) — **判断の根拠は全てここ。実装前に §4（主要な設計判断）を読むこと。**

## Global Constraints

- 言語: TypeScript ESM、import は拡張子 `.js` を明示。コメント/ログ/エラーは日本語。
- エラーは `@yancha/core` の `YanchaError`（code 付き）で送出。
- テスト: vitest（globals 有効）。外部I/O（LLM・YouTube API・ffmpeg）は差し替え可能にしてモック。
- **`Math.random()` / `new Date()` の直接使用を禁止**（seed駆動・決定論。設計 §4.1）。時刻は引数で注入する。
- **ffmpeg・音声・映像の実行は単体テストしない。** パラメータ→ffmpeg引数列の組み立てを純関数にしてテストする（設計 §6）。
- **P0スコープを超えない**（設計 §1 の非ゴール表）。雨1プリセット・シーン1種・60秒・限定公開まで。
- コミットは日本語 `prefix: 要約`。関連Issueを `#番号` で紐付ける。
- 開発は feature ブランチ → ローカルで `main` にマージ → `git push` → Issue更新（CLAUDE.md）。

- [ ] **Step 1: 429/5xx リトライ**

## Task 4: `packages/visual-synth` MVP

> **技術は実機検証で確定済み（設計 §7.2）: Puppeteer + headless Chrome + SwiftShader。**
> **`headless-gl` は採用しない**（GPUなしヘッドレス機で X display を要求し起動失敗。xvfb必須のため）。

**Files:**
- Add: `packages/visual-synth/{package.json,tsconfig.json,vitest.config.ts}`,
  `packages/visual-synth/src/{index.ts,browser.ts,server.ts,cli.ts}`,
  `packages/visual-synth/scenes/particles/{index.html,scene.js}`
- Test: `packages/visual-synth/tests/{sanity.test.ts,args.test.ts}`
- Modify: `src/stages/index.ts`, `src/stages/visual.ts`（新規）

**Interfaces:**
- Produces: `renderLoop(opts: { preset: "particles"; params; loopSeconds; seed; outPath }): Promise<void>`

- [ ] **Step 1: パッケージの雛形**

依存: `puppeteer`, `three`, `@yancha/core`。

- [ ] **Step 2: ⚠️ SwiftShader フラグを必ず明示する**

```js
args: ['--no-sandbox', '--disable-dev-shm-usage',
       '--use-gl=angle', '--use-angle=swiftshader-webgl', '--enable-unsafe-swiftshader']
```
- 値は **`swiftshader-webgl`**（`swiftshader` ではない）
- **フラグを落とすと例外なしで真っ白な動画ができる**（設計 §5.5）。Chrome 150 で自動フォールバック削除済み
- **このフラグ列を定数にまとめ、コメントで理由を書くこと**

- [ ] **Step 3: ローカルHTTP配信**

three を ESM で使うため importmap が必要 → **`file://` では動かない**。`node:http` で配信する。

- [ ] **Step 4: ⚠️ フレーム番号駆動にする（rAF禁止）**

`requestAnimationFrame` は壁時計時間に依存し**非決定論になる**（設計 §4.1 違反）。
→ `window.__renderFrame(frameIndex)` を `page.evaluate` から呼ぶ。

- [ ] **Step 5: 完全ループのシーン（パーティクル＋グラデーション）**

パーティクルの位相を `loopSeconds` で割り切れる周期にする（末尾フレーム＝先頭フレーム）。
**非フォトリアル様式**（上位設計 §2）。seed は `deriveSeed(videoId, "visual")`。

- [ ] **Step 6: フレーム取得は `page.screenshot`**

```js
await page.screenshot({ path: `frames/frame-${String(i).padStart(4,'0')}.png`, optimizeForSpeed: true });
```
**`readPixels`→base64→ffmpeg にしないこと**。実測で**遅い**（517ms/frame vs 372ms/frame）。
CDP越しのbase64転送コストがPNGエンコードより高い（設計 §7.2）。

- [ ] **Step 7: ⚠️ 白画面検出の assert（最重要）**

レンダリング直後に ffmpeg `signalstats` で輝度(YAVG)と分散を検査:
- YAVG が閾値外（真っ白=234.97 のような値）→ `YanchaError`
- 全フレームが同一サイズ／フレーム間の分散がゼロ → `YanchaError`

**これがないと壊れていることに誰も気付かない**（設計 §5.5）。**このタスクで最も重要なステップ。**

- [ ] **Step 8: ffmpeg で連番PNG → `visual-loop.mp4`**

**出力は尺全体ではなく `loopSeconds` のループ素材**（設計 §4.3）。伸長は Task 5 の `video` ステージ。

- [ ] **Step 9: 決定論の検証**

ブラウザプロセスを立ち上げ直して2回実行 → **フレームがバイト一致**することを確認
（実測で10/10一致を確認済み）。

- [ ] **Step 10: `VisualStage`**

`scene.json` を読んで `renderLoop` を呼ぶ。license追記＋生成ログ。

> **性能の目安（実測）**: 1080p/30fps で約2.7fps。**10秒ループ＝約2分**。
> 尺を伸ばしてもレンダリング時間は変わらない（設計 §4.3）。
