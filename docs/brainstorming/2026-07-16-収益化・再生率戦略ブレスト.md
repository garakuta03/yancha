## 0. 結論サマリ（先に要点）

1. **収益化条件のボトルネックは「登録者1,000人」であり、「4,000時間」ではない。**
   睡眠ニッチ＋ライブ配信なら総再生時間は容易に貯まる。戦略は
   「**時間はライブで稼ぎ、登録者は検索SEO動画とShortsで稼ぐ**」の二正面。
2. **毎晩の定時ライブ配信（22時〜翌7時）は Mac 完結の原則を守ったまま自動化できる**
   （launchd + ffmpeg RTMP push）。平均同接がわずか1人でも月270時間、5人なら月1,350時間。
   24/7ラジオ化（lofi girl型）はフェーズ1でサーバー化と同時に行う。
3. **睡眠ニッチはRPMが低い**（視聴者が寝ていて広告を見ない）。AdSense単体に頼らず、
   ①YouTube Premium按分（視聴時間ベース＝睡眠系に有利）、②同一アセットの
   Spotify/Apple Music/Podcast横展開（限界費用ほぼゼロ）、③将来の英語圏展開（RPM数倍）
   で補う。
4. **既存の research パッケージを「入口（テーマ選定）」に、新設の analytics 収集を
   「出口（実績フィードバック）」に接続した改善ループ**が、このシステム最大の自動化レバレッジ。
5. 広告は「**寝る前にだけ流す**」配置（冒頭〜15分のみ、以降ミッドロールなし）を
   ブランド方針として明示する。収益微減とひきかえに、低評価・登録解除を防ぎ
   「安心して眠れるチャンネル」という差別化資産になる。

## 1. 収益源の全体マップと現実的な単価感

| 収益源 | 単価感（目安） | 睡眠ニッチとの相性 | 自動化との相性 |
|---|---|---|---|
| AdSense（長尺動画） | RPM ¥100〜300程度と想定（低め） | △ 視聴者が寝るので広告主価値が低い | ◎ 本数×尺で積める |
| YouTube Premium按分 | 視聴時間比例 | ◎ 長時間・リピート視聴で有利 | ◎ 何もしなくても入る |
| ライブ配信（広告＋視聴時間） | 同上＋アーカイブ二重取り | ◎ 「つけっぱなし」需要と合致 | ◎ ffmpegループ配信 |
| Super Chat / メンバーシップ | YPP通過後 | △ 寝る前の投げ銭は期待薄。メンバー限定長尺版は可能性あり | ○ |
| 音楽ストリーミング配信（Spotify等） | 1再生 ¥0.3〜0.5 | ◎ 睡眠プレイリスト市場が大きい | ◎ 既存BGMアセットの再利用 |
| Podcast配信（朗読音声） | 広告/再生 | ○ 睡眠瞑想Podcastは定番ジャンル | ◎ narration.wavの再利用 |
| 企業案件（寝具・睡眠アプリ） | 後期 | ○ | × 人間対応 |

**設計含意**: 全アセットを自社ライセンスで持つ方針は、YouTube外への横展開（Spotify/Podcast）を
無条件で可能にする。これは「完全オリジナル主義」の隠れた収益的リターン。

## 2. 最重要インサイト — ライブ配信は収益化条件への最短路

### 2.1 数字で見る

- 収益化条件: 登録者1,000人 ＋ 直近12ヶ月4,000時間（= 月平均334時間）。
- 毎晩9時間配信（22:00〜7:00）: 平均同接1人で **月270時間**、同接5人で **月1,350時間**。
- 24/7配信（フェーズ1）: 平均同接5人で **月3,600時間**。4,000時間は1〜2ヶ月で到達圏。
- ライブのアーカイブを公開のまま残せば、VODとしても再生時間を稼ぐ（二重取り）。
- 一方、登録者1,000人はライブだけでは貯まりにくい → 検索SEO動画とShortsが受け持つ。

### 2.2 フェーズ0でもできる形（Mac完結の原則と両立）

24/7配信は常時稼働マシンが要るためフェーズ0の「Mac完結・サーバー依存なし」と緊張関係にある。
そこで段階を分ける:

| 段階 | 形態 | インフラ |
|---|---|---|
| フェーズ0後半 | **毎晩の定時配信**（22:00〜7:00、launchdで自動開始/終了） | MBP M2 Max（電源接続・スリープ無効化はスケジュール制御） |
| フェーズ1 | 24/7睡眠ラジオ | 安価なVPS or Linuxサーバー。事前レンダリング済みmp4を `-c copy` でRTMP push（再エンコード不要＝CPUほぼ食わない） |

技術的には `ffmpeg -re -stream_loop -1 -i loop.mp4 -c copy -f flv rtmp://a.rtmp.youtube.com/live2/<key>`
が骨格。**再エンコードしない**構成なら負荷・電気代とも無視できる水準。

### 2.3 ライブの「再利用コンテンツ判定」対策（＝差別化にもなる）

同一ループの垂れ流しはポリシーリスク。逆にここを自動化で作り込むと独自性になる:

- **プレイリストローテーション**: 自社BGMアセットをシャッフル・日替わり編成（playlist.jsonで管理）。
- **時刻連動の映像切替**: 配信時刻に応じて夜景→深夜→夜明けとシーンが変わる
  （ComfyUIで時間帯別素材を事前生成し、ffmpegの入力を切替）。「本物の夜と一緒に進む配信」は
  つけっぱなし視聴と相性がよく、他にない付加価値になる。
- **画面上の動的要素**: 現在の曲名・「あと○時間で夜明け」等のオーバーレイ（drawtext）。
- チャット対応などリアルタイム性はフェーズ2以降（自動化困難なので初期は捨てる）。

### 2.4 プレミア公開の活用（ライブ未満・動画以上）

新作動画は通常公開でなく**プレミア公開**にすると、待機ページ＋チャットで初速エンゲージメントが
付き、実質「ミニライブ」になる。API投稿時にフラグを立てるだけで自動化コストゼロ。
毎週同じ曜日・時刻（例: 金曜22時）に固定し「眠れない夜の定例」として習慣化を狙う。

## 3. 動画側 — 再生率（インプレッション→CTR→リテンション）の設計

### 3.1 流入は「検索・サジェスト・つけっぱなし自動再生」が主戦場

睡眠ニッチはブラウジング（トップページ）より検索・サジェスト比率が高い。打ち手:

- **キーワード駆動のテーマ選定**: research パッケージの keywords.yaml ＋ 競合エッジ指標を
  ステージ[0]の入力に直結する（→ §5）。「雨音 睡眠」「眠れない夜 朗読」等、
  検索ボリュームと競合の谷を突く。
- **タイトルの型**: 冒頭にキーワード＋尺表記＋用途。例「【睡眠導入】雨音と朗読で眠る夜 — 3時間」。
  効能断定は使わない（「不眠が治る」NG →「眠れない夜に」OK）。この言い換え辞書を
  メタデータ生成ステージ[7]のプロンプトに組み込む。
- **説明文にチャプター（タイムスタンプ）**: 台本生成[1]が構成（導入→呼吸誘導→ボディスキャン→情景）を
  持っているので、チャプターは**台本から機械的に生成できる**。検索結果のリッチ表示にも効く。
- **シリーズ化＋再生リスト**: 「終わったら次が自動再生される」構造が睡眠系のセッション時間を
  最大化する。動画は必ずシリーズ（例: 森の夜シリーズ）に属させ、投稿時にAPIで再生リストへ追加。
  エンドスクリーンは寝ていて見ないので、**自動再生の連鎖設計のほうが重要**。

### 3.2 リテンションカーブの特殊性を利用する

睡眠動画の視聴維持は「最初の1〜3分で寝るか離脱するかが決まり、その後は寝ているので平坦」という
特異な形になる。つまり**勝負は冒頭60秒**:

- 大音量イントロ・自己紹介・チャンネル登録のお願いを冒頭に置かない。即、静かな価値提供に入る。
- 冒頭30秒の台本テンプレートを固定化し、リテンション実測（§5のanalytics）でテンプレ自体をA/B改善。
- 音量ノーマライズ（ラウドネス -20〜-16 LUFS程度の睡眠向け基準）をミックスステージ[4]の
  自動チェックに入れる。「急に大きい音がした」は睡眠系で最悪の離脱・低評価要因。

### 3.3 サムネイル・タイトルのA/Bテスト

- YouTube公式の「テストと比較（サムネA/B）」を毎本使う。サムネ生成を**常に3案**出す
  パイプラインにし、人間レビュー[8]で3案とも承認→A/Bに投入。
- 睡眠系サムネの定石: 暗色ベース・単一モチーフ・大きい文字少なめ・青/紫系。
  夜のフィードで「眩しいサムネ」は忌避される。

### 3.4 Shorts — 登録者獲得ファネル

- 長尺から60秒切り出し（「1分だけ呼吸を整える」等）＋ 縦組みテロップ。final.mp4 と script.md から
  **派生ステージとして完全自動化可能**。
- Shorts のRPMは低いが目的は収益でなく**登録者1,000人の突破**と長尺への導線。
- 1本の長尺から2〜3本のShortsを機械生成 → 人間レビューで選別、が費用対効果最良。

### 3.5 多言語展開（フェーズ1以降の最大レバー）

- 台本→英訳→英語TTSは既存パイプラインの分岐1本で済む。英語圏の睡眠ニッチはRPMが数倍。
- 選択肢: (a) 同一動画にマルチ音声トラック、(b) 英語別チャンネル。
  アルゴリズム最適化の観点は(b)が有利だが、登録者要件を二重に負う。フェーズ0では着手しない。

## 4. 広告配置とブランド戦略 — 「寝る前にだけ広告」

- ミッドロールを機械任せにすると就寝後（30分以降）に音声広告が入り、視聴者を起こす。
  これは低評価・登録解除・「二度と来ない」に直結する。
- 方針: **プレロール＋冒頭15分以内のミッドロールのみ。以降は広告なし**。概要欄にも明記
  （「◯分以降に広告は入りません」）。収益は数%落ちるが、信頼がリピート視聴＝Premium按分と
  登録者を生む。ミッドロール位置はAPI/Studioで手動指定できるため、位置候補を
  メタデータ生成[7]が算出→人間レビューで確定、と半自動化できる。

## 5. 自動化への落とし込み — 既存システムとの接続

### 5.1 改善ループの全体像

```
research（既存）                      analytics（新設）
 キーワード×競合エッジ指標   ┌──  自チャンネル実績（リテンション・流入元・CTR）
        │                    │
        ▼                    ▼
[0] テーマ選定 ← 週次「テーマ候補トップN」レポート（人間が選ぶだけの状態にする）
        ▼
[1]〜[7] 既存パイプライン（＋派生: Shorts切り出し / チャプター生成 / サムネ3案）
        ▼
[8] 人間レビュー（サムネ3案承認・広告位置確定・効能表現チェック）※自動化しない
        ▼
[9] 投稿（プレミア公開フラグ・再生リスト追加・投稿時刻キュー 21〜23時）
        ▼
[10] ライブ運転（新設 packages/live: 定時配信 → 将来24/7）
```

### 5.2 新設・拡張するコンポーネント

| コンポーネント | 内容 | フェーズ |
|---|---|---|
| `packages/analytics` | YouTube Analytics API（OAuth）で自チャンネルのリテンション・流入元・CTRを収集し、researchの競合指標と同じWebビューに並べる | 0後半〜1 |
| `packages/live` | playlist.json駆動のffmpeg RTMP push。launchd定時起動→将来Docker化してVPS/Linuxへ | 0後半（定時版）／1（24/7版） |
| Shorts派生ステージ | final.mp4＋script.mdから縦型60秒を2〜3案生成 | 0（半自動でも可） |
| チャプター生成 | script.mdの構成見出し→説明文タイムスタンプ | 0（実装コスト極小） |
| 投稿キュー | 人間承認済みキューから最適時刻にAPI投稿・再生リスト追加・プレミア設定 | 1 |
| 効能表現リンター | NG表現辞書（「治る」「改善」「〇〇Hzで…」等）でタイトル・説明・台本を機械チェックし、人間レビューの負荷を下げる | 0（実装コスト極小） |

### 5.3 既存researchパッケージの活かし方

- `collect` の競合エッジ指標から「**伸びているのに投稿頻度が低い**」フォーマットを特定する
  ビュー（＝需要過多・供給不足の谷）を追加。テーマ選定の一次情報にする。
- `discover` のキーワード台帳は、そのままタイトルSEO語彙の供給源として[7]にも渡す。

## 6. AdSense外の横展開（アセット再利用・限界費用ほぼゼロ）

1. **Spotify / Apple Music / YouTube Music**: 自社BGMをDistroKid等で配信。睡眠プレイリストは
   ストリーミング市場でも大きい。全曲自社ライセンスなので障害なし。証跡管理（license.json）が
   ここでも効く。
2. **Podcast**: narration.wav＋BGMミックスをそのまま「睡眠導入ガイド」Podcastとして
  Spotify/Apple Podcastsに配信。RSS生成は自動化容易。
3. 将来: メンバーシップ限定の長尺版（3時間→8時間版はffmpegループで生成コストほぼゼロ）。

## 7. リスク・原則整合チェック

| 打ち手 | 抵触しうる原則/ポリシー | 対策 |
|---|---|---|
| 24/7ライブ | 再利用・繰り返しコンテンツ | ローテーション・時刻連動シーン・動的オーバーレイで毎晩異なる配信にする（§2.3） |
| 定時配信の自動開始 | スパム/自動化ポリシー | 配信「内容」は人間レビュー済みアセットのみ。編成変更時は人間承認を挟む |
| Shorts量産 | 薄いコンテンツ量産の禁止 | 1長尺→2〜3本まで。独自台本由来なので転載型Shortsとは質的に異なる |
| SEOタイトル | 効能の誇大表示 | 効能表現リンター＋言い換え辞書（§3.1・§5.2） |
| フェーズ0のVPS利用 | Mac完結の原則 | フェーズ0は定時配信（Mac）に留め、24/7サーバー化はフェーズ1で解禁 |
| 多言語・別チャンネル | フェーズ0のYAGNI | フェーズ1以降に明示的に後ろ倒し |

## 8. 優先順位つきロードマップ（費用対効果順）

### フェーズ0（今〜勝ち筋特定まで）— 追加コストほぼゼロの施策のみ

1. チャプター生成・効能表現リンター・サムネ3案化（実装小・効果確実）
2. タイトルSEOの型を research キーワードに接続（既存資産の接続のみ）
3. 動画5本たまった時点で**毎晩の定時配信を開始**（launchd＋ffmpeg。総再生時間の主エンジン）
4. Shorts切り出しを半自動で開始（登録者ファネル）
5. プレミア公開を毎週定時で運用（コストゼロ）
6. analytics収集を最小実装し、冒頭60秒テンプレのA/B改善を回し始める

### フェーズ1（勝ち筋特定後）

1. 24/7睡眠ラジオのサーバー化（時刻連動シーン切替つき）
2. 投稿キュー・A/B・分析ループの完全自動化（人間レビューは維持）
3. Spotify/Podcast横展開
4. 英語展開（別チャンネル）— RPM数倍の最大レバー

### 収益の目安（保守的シナリオ）

- 〜3ヶ月: 収益ゼロ（検証期）。定時配信開始で総再生時間の貯蓄開始。
- 3〜6ヶ月: 登録者1,000人が主戦。Shorts＋検索SEOで突破 → YPP申請。
  4,000時間はライブで先に満たしている想定。
- 6〜12ヶ月: AdSense＋Premium按分で月数千〜数万円圏。横展開・英語圏で逓増を狙う。
  （睡眠ニッチのRPMは低いため、「本数×尺×ライブ×横展開」の総和で稼ぐ構造が前提）

## 9. 未決事項（次の壁打ち候補）

- 定時配信の編成単位（毎晩同一テーマか、曜日替わりか）
- プレミア公開の固定枠（曜日・時刻）
- analytics のOAuth運用（API keyでは不可・要Googleアカウント連携）の置き場所
- Spotify配信のディストリビューター選定（DistroKid / TuneCore / その他）
